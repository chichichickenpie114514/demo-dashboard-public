Data Lineage — From Raw Claim to Dashboard
1. Raw billing_records
Each row: base_name + sales_type + credit_amount + person_id + debit_account + service_month. Source: MySQL Cloud SQL, pre-aggregated monthly.
2. display_for_base_name() — NFKC Normalization
Full-width/half-width unification, whitespace normalization. Rule-based name deduplication merges variant spellings of the same facility. Unresolvable rows → NULL (excluded from aggregates).
3. service_office Resolution + Category Classification
Normalized base_name → office_name → area. sales_type → category via pattern matching. sales_type → is_residential (bool) — residential care types detected by keyword patterns with NFKC normalization.
Branches into two aggregation paths
Path A: Monthly Summary
Aggregated by service_office × sales_type. Feeds 拠点, 分析, サービス, 利用者, 推移, マップ tabs. SUM(credit_amount) GROUP BY. Residential users identified by is_residential flag.
Path B: Journal Entry
From journal_entries at accounting line granularity. Feeds 拠点(配賦) and 拠点(配賦2) tabs. Split-aware facility resolution. Haifu matrix + management journal overlay.

Path A and Path B totals agree within 0.2%. Variance from proration rounding and zero-net row removal.

システムプロンプト (docs/AI_CHAT_PROMPT.md) は 8 タブ・データモデル・配賦ロジックの全詳細を含む。streaming + thinking indicator + elapsed timer。marked.js による Markdown レンダリング。文脈に応じたフォローアップ提案。

Filter Awareness: 16 の分析 scope と 8 つの chat tools すべてが、現在の期間選択 (start/end) と UI フィルター状態を AI に渡す。chat のシステムプロンプトはダッシュボードアーキテクチャの全文を含むため、「この数字の意味は？」と「このダッシュボードどう動いてるの？」の両方に答えられる。

社内間配賦 (Internal Allocation)

Business Logic

訪問介護・訪問看護のステーションは居住系施設にスタッフを派遣する。保険請求はステーションが行うが、実際のサービス提供は居住施設で発生する。配賦システムはステーションの売上を、スタッフが実際に働いた居住系施設に再分配する。

居住系施設 → 流入 (+) — ステーションから配分される売上。緑色表示。

訪問ステーション → 流出 (−) — 居住施設へ配分する売上。赤色表示。

∑ inflow + ∑ outflow = 0 — 配賦は全社レベルで収支中立。

Implementation

オリジナルは Google Sheets の配賦マトリクスから haifu_sync.py で読み込み。月次・売上コード別に配賦先（destinations, +）と配賦元（sources, −）を記録。

2 つのカードビュータブが配賦データを重畳:

  • 拠点（配賦） — 請求 + 配賦の合算
  • 拠点（配賦2） — 請求 + 配賦 + 管理会計仕訳（5 つの非標準売上コード）

デモ版では決定論的にバランスの取れた配賦ペアを生成（haifu_sync.py の fetch() は空リストを返し、generate_demo_data.py が合成データを作成）。
